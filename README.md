# SchulyDocs

Unified [Docusaurus](https://docusaurus.io/) site that aggregates the documentation of
every repo in the [`schulydev`](https://github.com/schulydev) org into one searchable,
cross-linkable site.

Each source repo keeps its docs next to its code in a `docs/` folder. A GitHub Actions
workflow in each repo syncs that folder into `docs/<Repo>/` here on every push to `main`,
and [Cloudflare Pages](https://pages.cloudflare.com/) rebuilds the site automatically.

> **You usually don't edit docs in this repo.** Edit `docs/` in the source repo; it
> shows up here within a minute. The only things owned here are the site config and each
> section's title/order (`docs/<Repo>/_category_.json`).

## Architecture

```
source repos (Schuly, SchulyBackend, ...)        this repo (SchulyDocs)        Cloudflare Pages
  docs/**  --push to main-->  GH Action  --commit-->  docs/<Repo>/  --push-->  rebuild & deploy
```

- **One** Docusaurus site, **one** docs plugin instance. Every repo's docs live under
  `docs/<Repo>/`, giving unified search and free cross-linking between sections.
- Section labels and ordering are controlled in `docs/<Repo>/_category_.json` — files
  this repo owns. The sync **preserves** them, so re-syncing a source repo never
  clobbers a section's title or position. Everything else in `docs/<Repo>/` is replaced
  on each sync (so deletions in the source propagate).
- The sidebar (`sidebars.js`) is fully autogenerated from the `docs/` tree. Each repo's
  `README.md` is automatically used as that section's landing page.

## Local development

Uses **bun** (not npm/node scripts directly).

```bash
bun install
bun run start        # dev server at http://localhost:3000
bun run build        # production build into build/
bun run serve        # serve the production build locally
```

To preview with the latest docs from sibling repos cloned next to this one:

```bash
bash scripts/sync-docs.sh
bun run start
```

## Hosting: Cloudflare Pages (free tier)

Cloudflare Pages has native GitHub integration and an official Docusaurus preset, so
hosting needs no CI config on the Cloudflare side — it watches this repo and rebuilds on
every push.

1. Cloudflare dashboard → **Workers & Pages** → **Create** → **Pages** →
   **Connect to Git** → pick `schulydev/SchulyDocs`.
2. Framework preset: **Docusaurus**. It fills in:
   - Build command: `npx docusaurus build` (or set `bun run build`)
   - Build output directory: `build`
   - To force bun: add an env var `BUN_VERSION` and keep `bun.lock` committed; Cloudflare
     detects bun from the lockfile.
3. Deploy. You get a `*.pages.dev` URL. Add a custom domain later under the project's
   **Custom domains** tab (and update `url` in `docusaurus.config.js`).

From then on: any push here (including the automated doc-sync commits) triggers a rebuild.

## Wiring up a source repo (one-time, per repo)

1. Copy [`templates/sync-docs.yml`](templates/sync-docs.yml) into the source repo at
   `.github/workflows/sync-docs.yml`.
2. Create a fine-grained PAT (or GitHub App token) with **Contents: Read & Write** scoped
   to `schulydev/SchulyDocs` only, and add it as a secret named `DOCS_SYNC_TOKEN`. Tip:
   make it an **organization** secret scoped to the source repos so you set it once.
3. Push a change under `docs/` (or run the workflow manually) — the section appears here.

To add a brand-new repo's section: also add `docs/<Repo>/_category_.json` here with the
desired `label` and `position`, and add the repo to the list in `scripts/sync-docs.sh`.

## Notes / limitations

- Source docs are plain GitHub-flavored Markdown, so `.md` is parsed as CommonMark
  (`markdown.format: 'detect'` in the config) to avoid MDX choking on raw `<...>` text.
- Nested folders (e.g. `setup/`) become sub-categories labeled by folder name. To rename
  one, add a `_category_.json` inside that folder **in the source repo** (it travels with
  the docs).
- Cross-section links should use absolute site paths (e.g. `/SchulyBackend/architecture`)
  or full GitHub URLs; relative links only resolve within the same section.
